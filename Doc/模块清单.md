### 阶段 1：补充 P0 级设计（本周完成）

#### 任务 1.1：SQL 函数系统架构设计（1 天）

**设计目标**：为已实现的 60+ 函数提供架构文档，明确扩展机制

**设计内容**：
1. **函数分类体系**
   - 聚合函数（COUNT/SUM/AVG/MIN/MAX/STRING_AGG 等）
   - 标量函数（字符串/数值/日期/转换/条件/系统/哈希/地理向量）
   - 表值函数（未来扩展）

2. **函数注册机制**
   - 内置函数表（FunctionRegistry）
   - 函数签名（名称、参数类型、返回类型）
   - 函数实现（委托/表达式树）

3. **执行流程**
   - SQL 解析 → 函数名识别 → 参数类型检查 → 委托调用
   - 聚合函数：分组 → 累积 → 计算
   - 标量函数：逐行计算

4. **扩展点设计**
   - 用户自定义函数（UDF）接口
   - 注册 API（`SqlEngine.RegisterFunction(name, func)`）

5. **内置函数清单**
   - 引用《SQL函数支持清单.md》作为附录
   - 补充每个函数的实现类/方法位置

### 阶段 2：补充 P1 级设计（下周完成）

#### 任务 2.1：自动分片架构设计（0.5 天）

**设计目标**：为 ShardManager 提供详细的分片策略文档

**设计内容**：
1. **分片触发条件**
   - 单分片行数阈值（默认 100 万行）
   - 单分片文件大小阈值（默认 1GB）
   - 手动触发分片命令

2. **分片文件命名规则**
   - `{TableName}_{ShardId}.data`（例如：`users_0.data`、`users_1.data`）
   - ShardId 从 0 开始递增

3. **路由算法**
   - 写入路由：写入最新未满分片
   - 读取路由：遍历所有分片，合并结果
   - 主键查询优化：根据 ShardInfo.MinKey/MaxKey 定位分片

4. **分片元数据**
   - ShardInfo：ShardId、RowCount、FileSize、IsReadOnly、MinKey、MaxKey
   - 存储位置：nova.db 元数据文件

5. **分片生命周期**
   - 创建：达到阈值时自动创建新分片
   - 归档：旧分片标记为只读
   - 删除：不支持删除单个分片（避免主键空洞）

**交付物**：

## 六、架构设计审查流程

### 审查标准

每个模块的架构设计需包含：
1. ✅ **功能边界**：明确模块职责与不做什么
2. ✅ **数据结构**：核心类/接口定义（字段、方法、依赖）

4. ✅ **设计决策**：技术选型理由（如：为什么用 SkipList 而非 B+ 树）
5. ✅ **性能指标**：关键路径的性能目标（如：主键查询 < 1ms）
6. ✅ **测试策略**：正常/边界/异常场景覆盖

```
AI 起草设计 → 人工审查 → 讨论修改 → 定稿 → 合并到架构设计文档
```

**审查人**：NewLife 核心开发者（至少 1 人）
