# NewLife.NovaDb - 集群架构

## 1. 概述

NovaDb 集群采用**一主多从**架构,通过 Binlog 流复制实现主从同步,支持断点续传与最终一致性。

## 2. 架构拓扑

```
┌───────────────┐
│  Master Node  │  ← 主节点（读写）
│  (可写可读)    │     - 处理所有写操作
└───────┬───────┘     - 写入 Binlog
        │
        │ Binlog Stream
        ├─────────────────┐
        ↓                 ↓
┌───────────────┐  ┌───────────────┐
│ Replica Node 1│  │ Replica Node 2│  ← 从节点（只读）
│  (只读)        │  │  (只读)        │     - 接收 Binlog 流
└───────────────┘  └───────────────┘     - 重放逻辑操作
                                          - 提供读服务
```

### 2.1 拓扑特性

- **一主多从**：单个主节点,多个从节点
- **主节点**：处理所有写操作,写入 Binlog
- **从节点**：接收 Binlog 流,重放操作,提供只读服务
- **扩展性**：从节点数量可水平扩展,分担读压力

## 3. 核心组件

### 3.1 ReplicationManager（主节点）

复制管理器,负责 Binlog 流发送:

```csharp
public class ReplicationManager
{
    private readonly List<ReplicaClient> _replicas = [];
    private readonly BinlogReader _binlogReader;
    
    /// <summary>注册从节点</summary>
    public void RegisterReplica(ReplicaClient replica)
    {
        _replicas.Add(replica);
    }
    
    /// <summary>广播 Binlog 记录</summary>
    public void BroadcastBinlogRecord(BinlogRecord record)
    {
        foreach (var replica in _replicas)
        {
            try
            {
                replica.SendBinlog(record);
            }
            catch (Exception ex)
            {
                Log?.Error($"发送 Binlog 到从节点失败: {replica.NodeId}, {ex.Message}");
                // 标记从节点为异常状态,稍后重试
                replica.MarkAsUnhealthy();
            }
        }
    }
    
    /// <summary>处理从节点的同步请求</summary>
    public List<BinlogRecord> GetBinlogsSince(Int64 lsn)
    {
        return _binlogReader.ReadSince(lsn);
    }
}
```

### 3.2 ReplicaClient（从节点）

从节点客户端,负责 Binlog 流接收与重放:

```csharp
public class ReplicaClient
{
    private readonly NovaClient _masterClient;
    private Int64 _lastSyncedLsn = 0;
    
    /// <summary>连接到主节点</summary>
    public void ConnectToMaster(String masterAddress, Int32 port)
    {
        _masterClient = new NovaClient
        {
            Server = masterAddress,
            Port = port
        };
        
        _masterClient.Connect();
    }
    
    /// <summary>开始同步 Binlog</summary>
    public async Task StartReplicationAsync()
    {
        while (!_cancellationToken.IsCancellationRequested)
        {
            try
            {
                // 1. 请求新的 Binlog 记录（从 _lastSyncedLsn 开始）
                var records = await _masterClient.GetBinlogsSinceAsync(_lastSyncedLsn);
                
                // 2. 重放 Binlog 记录
                foreach (var record in records)
                {
                    ReplayBinlogRecord(record);
                    _lastSyncedLsn = record.Lsn;
                }
                
                // 3. 如果没有新记录,短暂休眠
                if (records.Count == 0)
                {
                    await Task.Delay(1000, _cancellationToken);
                }
            }
            catch (Exception ex)
            {
                Log?.Error($"Binlog 同步失败: {ex.Message}");
                await Task.Delay(5000, _cancellationToken); // 失败后等待 5 秒重试
            }
        }
    }
    
    /// <summary>重放 Binlog 记录</summary>
    private void ReplayBinlogRecord(BinlogRecord record)
    {
        using var tx = _localEngine.BeginTransaction();
        
        switch (record.Type)
        {
            case BinlogRecordType.Insert:
                _localEngine.Execute($"INSERT INTO {record.TableName} (...) VALUES (...)", tx);
                break;
            
            case BinlogRecordType.Update:
                _localEngine.Execute($"UPDATE {record.TableName} SET ... WHERE ...", tx);
                break;
            
            case BinlogRecordType.Delete:
                _localEngine.Execute($"DELETE FROM {record.TableName} WHERE ...", tx);
                break;
            
            case BinlogRecordType.CreateTable:
                _localEngine.Execute(record.DdlSql!, tx);
                break;
            
            case BinlogRecordType.DropTable:
                _localEngine.Execute($"DROP TABLE {record.TableName}", tx);
                break;
        }
        
        tx.Commit();
    }
}
```

### 3.3 NodeInfo（节点信息）

```csharp
public class NodeInfo
{
    public String NodeId { get; set; } = Guid.NewGuid().ToString();
    public String Endpoint { get; set; } = ""; // "192.168.1.100:3306"
    public NodeRole Role { get; set; } = NodeRole.Replica; // Master / Replica
    public Int64 LastSyncedLsn { get; set; } = 0; // 最后同步的 LSN
    public DateTime LastHeartbeat { get; set; } = DateTime.Now;
    public NodeStatus Status { get; set; } = NodeStatus.Healthy; // Healthy / Unhealthy / Offline
}

public enum NodeRole
{
    Master,
    Replica
}

public enum NodeStatus
{
    Healthy,   // 正常
    Unhealthy, // 异常（同步延迟、连接失败）
    Offline    // 离线
}
```

## 4. 复制流程

### 4.1 完整同步流程

```
初始化阶段:
  1. 从节点启动
  2. 连接到主节点
  3. 读取本地最后同步的 LSN（_lastSyncedLsn）
  4. 如果 LSN = 0（首次同步）:
     a. 请求全量数据快照（可选）
     b. 或从 LSN = 0 开始增量同步
  5. 开始增量同步

增量同步阶段:
  1. 从节点向主节点请求: "给我 LSN > {_lastSyncedLsn} 的 Binlog"
  2. 主节点返回新的 Binlog 记录列表
  3. 从节点逐条重放 Binlog 记录
  4. 更新 _lastSyncedLsn
  5. 循环回到步骤 1
```

### 4.2 断点续传

```csharp
// 从节点启动时加载上次同步位置
private void LoadLastSyncedLsn()
{
    // 从元数据文件读取
    var metaFile = Path.Combine(_dbPath, ".replication.meta");
    if (File.Exists(metaFile))
    {
        var json = File.ReadAllText(metaFile);
        var meta = JsonSerializer.Deserialize<ReplicationMeta>(json);
        _lastSyncedLsn = meta.LastSyncedLsn;
    }
}

// 同步完成后保存位置
private void SaveLastSyncedLsn()
{
    var meta = new ReplicationMeta { LastSyncedLsn = _lastSyncedLsn };
    var json = JsonSerializer.Serialize(meta);
    File.WriteAllText(Path.Combine(_dbPath, ".replication.meta"), json);
}
```

### 4.3 主节点写流程

```
1. 客户端发起写事务（INSERT/UPDATE/DELETE）
2. 主节点执行事务 → 写入 WAL
3. 事务提交 → 写入 Binlog
4. ReplicationManager 广播 Binlog 记录给所有从节点
5. 从节点异步接收与重放（不阻塞主节点提交）
```

## 5. 一致性模型

### 5.1 最终一致性（默认）

- 主节点写入后立即返回,不等待从节点同步
- 从节点异步重放 Binlog,存在同步延迟（通常毫秒级）
- 适合读多写少场景,性能最优

### 5.2 强一致性读（可选）

```csharp
/// <summary>从主节点读取（强一致性）</summary>
public List<Dictionary<String, Object?>> ReadFromMaster(String sql)
{
    return _masterClient.Query(sql);
}

/// <summary>从从节点读取（最终一致性）</summary>
public List<Dictionary<String, Object?>> ReadFromReplica(String sql)
{
    return _replicaEngine.Query(sql);
}
```

### 5.3 同步延迟监控

```csharp
/// <summary>获取从节点滞后量（延迟的 LSN 数量）</summary>
public Int64 GetReplicationLag()
{
    var masterLsn = _masterClient.GetCurrentLsn();
    return masterLsn - _lastSyncedLsn;
}
```

## 6. 故障处理

### 6.1 从节点故障

- **检测**: 主节点心跳超时或 Binlog 发送失败
- **处理**: 标记从节点为 `Unhealthy`,停止发送 Binlog
- **恢复**: 从节点重启后,从 `_lastSyncedLsn` 继续同步

### 6.2 主节点故障（未来扩展）

- **检测**: 从节点心跳超时
- **选主**: 从节点发起选举,选出新主节点（基于 LSN 最大者）
- **切换**: 新主节点接管写操作,其他从节点同步到新主节点

### 6.3 网络分区

- **脑裂预防**: 使用 Quorum（多数派）机制,避免双主
- **数据修复**: 网络恢复后,从节点从断点继续同步

## 7. 性能优化

### 7.1 并行重放

```csharp
// 多线程并行重放（非冲突事务）
var parallelTasks = records
    .GroupBy(r => r.TableName) // 按表分组
    .Select(group => Task.Run(() =>
    {
        foreach (var record in group)
        {
            ReplayBinlogRecord(record);
        }
    }));

await Task.WhenAll(parallelTasks);
```

### 7.2 批量重放

```csharp
// 批量重放（减少事务开销）
using var tx = _localEngine.BeginTransaction();

foreach (var record in records.Take(1000))
{
    ReplayBinlogRecord(record, tx);
}

tx.Commit();
```

### 7.3 Binlog 压缩

- 主节点发送前压缩 Binlog（LZ4、Snappy）
- 从节点接收后解压
- 减少网络传输量

## 8. 设计决策

### D1: 一主多从拓扑
- 简化一致性协议,无需分布式共识
- 主节点处理所有写,从节点分担读
- 适合读多写少场景

### D2: Binlog 流复制
- Binlog 是逻辑日志,跨版本兼容性好
- WAL 是物理日志,仅用于本地崩溃恢复

### D3: 异步复制（最终一致性）
- 主节点不等待从节点,性能最优
- 从节点故障不影响主节点写入

### D4: 断点续传
- 从节点持久化 `_lastSyncedLsn`
- 崩溃恢复后从断点继续同步,避免全量重建

### D5: Binlog 独立于 WAL
- WAL 用于本地持久性,Binlog 用于跨节点复制
- 职责分离,互不干扰

## 9. 未来扩展

### 9.1 自动故障转移（Failover）
- 主节点故障时,自动选举新主节点
- 集成 ZooKeeper / etcd 进行选主协调

### 9.2 半同步复制
- 主节点等待至少一个从节点同步后才返回
- 平衡性能与一致性

### 9.3 级联复制
- 从节点可作为其他从节点的主节点
- 减少主节点网络压力

### 9.4 读写分离中间件
- 透明路由读请求到从节点,写请求到主节点
- 支持负载均衡与故障切换
