# NewLife.NovaDb - 核心层架构

## 1. 概述

核心层 (Core) 提供了 NovaDb 的基础设施，包括数据类型系统、数据编解码、异常处理、配置选项等。所有上层模块（存储、WAL、事务、引擎等）都依赖核心层。

## 2. 数据类型系统 (`DataType`)

### 2.1 基础数据类型

NovaDb 支持 8 种基础数据类型，严格映射 C# 类型：

| DataType | C# 类型 | 存储大小 | 说明 |
|----------|---------|---------|------|
| Boolean | Boolean | 1 字节 | 真/假 |
| Int32 | Int32 | 4 字节 | 32 位整数 |
| Int64 | Int64 | 8 字节 | 64 位整数 |
| Double | Double | 8 字节 | 双精度浮点数 |
| Decimal | Decimal | 16 字节 | 高精度十进制 |
| String | String | 变长 | UTF-8 编码，带长度前缀 |
| ByteArray | Byte[] | 变长 | 字节数组，带长度前缀 |
| DateTime | DateTime | 8 字节 | 日期时间，存储为 Ticks |

### 2.2 扩展数据类型

#### 2.2.1 GeoPoint（地理坐标）

- **目标类型**：`GEOPOINT`
- **C# 映射**：自定义结构体 `GeoPoint { Double Lat; Double Lon; }`
- **存储大小**：16 字节（两个 Double）
- **常用函数**：
  - `DISTANCE(point1, point2)`：两点距离（米），使用 Haversine 公式
  - `WITHIN_RADIUS(point, center, radius)`：是否在半径范围内
- **索引方向**：可选 GeoHash 或 R-Tree 空间索引，用于范围查询（如 BoundingBox）与附近查询
- **适用场景**：LBS 应用、附近商家/用户查询、地理围栏

#### 2.2.2 Vector（向量）

- **目标类型**：`VECTOR(n)`（定长向量，n 为维度）
- **C# 映射**：`Single[]`（长度固定为 `n`）
- **存储大小**：`n * 4` 字节（每个 Single 占 4 字节）
- **常用函数**：
  - `COSINE_SIMILARITY(vec1, vec2)`：余弦相似度（-1 到 1）
  - `EUCLIDEAN_DISTANCE(vec1, vec2)`：欧氏距离
  - `DOT_PRODUCT(vec1, vec2)`：点积
- **索引方向**：可选 HNSW（层次可导航小世界图）或 IVF（倒排文件）等 ANN 索引，用于 KNN 查询
- **适用场景**：语义搜索、推荐系统、RAG（检索增强生成）

## 3. 数据编解码 (`IDataCodec`)

### 3.1 接口定义

```csharp
public interface IDataCodec
{
    Byte[] Encode(Object? value, DataType type);
    Object? Decode(Byte[] data, DataType type);
}
```

### 3.2 默认编解码器 (`DefaultDataCodec`)

| 类型 | 编码规则 | 解码规则 |
|------|---------|---------|
| NULL | -1 长度前缀（4 字节） | 检测到 -1 长度 → 返回 null |
| Boolean | 1 字节（0/1） | 读取 1 字节，转换为 bool |
| Int32 | 4 字节小端序 | 读取 4 字节，转换为 Int32 |
| Int64 | 8 字节小端序 | 读取 8 字节，转换为 Int64 |
| Double | 8 字节（IEEE 754） | 读取 8 字节，转换为 Double |
| Decimal | 16 字节（128 位） | 读取 16 字节，转换为 Decimal |
| String | 4 字节长度 + UTF-8 字节 | 读取长度 → 读取 UTF-8 字节 → 转字符串 |
| ByteArray | 4 字节长度 + 字节数组 | 读取长度 → 读取字节数组 |
| DateTime | 8 字节（Ticks） | 读取 8 字节 → 转 DateTime |

### 3.3 编码示例

```csharp
// 编码字符串 "hello"
// 1. 转 UTF-8: [104, 101, 108, 108, 111] (5 字节)
// 2. 写长度前缀: [5, 0, 0, 0] (小端序 Int32)
// 3. 写内容: [104, 101, 108, 108, 111]
// 最终: [5, 0, 0, 0, 104, 101, 108, 108, 111] (9 字节)

// 编码 NULL
// 长度前缀 -1: [255, 255, 255, 255] (4 字节)
```

## 4. 异常与错误码 (`NovaException`)

### 4.1 异常类型

所有 NovaDb 异常均继承自 `NovaException`：

```csharp
public class NovaException : Exception
{
    public Int32 ErrorCode { get; }
    
    public NovaException(Int32 errorCode, String message)
        : base(message)
    {
        ErrorCode = errorCode;
    }
}
```

### 4.2 错误码分段

| 错误码范围 | 模块 | 示例 |
|-----------|------|------|
| 1000-1999 | 文件/存储错误 | 1001: 文件损坏, 1002: 校验和不匹配 |
| 2000-2999 | 解析/语法错误 | 2001: SQL 语法错误, 2002: 不支持的 SQL 语句 |
| 3000-3999 | 事务错误 | 3001: 事务已提交, 3002: 事务冲突 |
| 4000-4999 | 表/分片/流/KV 错误 | 4001: 表不存在, 4002: 列类型不匹配 |
| 5000-5999 | 不支持的操作 | 5001: 不支持的数据类型, 5002: 不支持的索引类型 |
| 6000-6999 | I/O 错误 | 6001: 磁盘空间不足, 6002: 文件读取失败 |
| 7000-7999 | 连接/协议错误 | 7001: 连接超时, 7002: 协议版本不匹配 |
| 8000-8999 | 复制/集群错误 | 8001: 主节点连接失败, 8002: Binlog 同步错误 |

### 4.3 使用示例

```csharp
// 抛出异常
if (table == null)
    throw new NovaException(ErrorCode.TableNotFound, $"表不存在: {tableName}");

// 捕获异常
try
{
    engine.Execute("SELECT * FROM unknown_table");
}
catch (NovaException ex)
{
    Console.WriteLine($"错误码: {ex.ErrorCode}, 消息: {ex.Message}");
}
```

## 5. 数据库选项

### 5.1 基础选项 (`DbOptions`)

嵌入模式与网络模式共用的基础配置：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|-------|------|
| Path | String | - | 数据库文件夹路径 |
| WalMode | WalMode | Normal | WAL 模式（Full/Normal/None） |
| PageSize | Int32 | 4096 | 页面大小（字节） |
| PageCacheSize | Int32 | 1024 | 页面缓存大小（页数） |
| HotWindowSeconds | Int32 | 600 | 热数据窗口（秒），超过此时间未访问则降为冷数据 |
| ColdEvictionSeconds | Int32 | 1800 | 冷数据淘汰时间（秒），超过此时间未访问则从内存卸载 |
| EnableChecksum | Boolean | true | 是否启用校验和验证 |
| ReadOnly | Boolean | false | 是否只读模式（禁止写操作，跳过 WAL 初始化，提升读取性能） |

### 5.2 WAL 模式说明

| WalMode | 持久性 | 性能 | 说明 |
|---------|-------|------|------|
| Full | 最强 | 最慢 | 每次写事务同步刷盘（fsync），崩溃后零数据丢失 |
| Normal | 适中 | 适中 | 准实时异步刷盘，崩溃后可能丢失最后几秒数据 |
| None | 无保证 | 最快 | 不写 WAL，崩溃后数据无法恢复 |

### 5.3 只读模式说明

- **适用场景**：数据分析、历史数据查询、备份节点只读访问
- **性能优势**：
  - 跳过 WAL 初始化与刷盘逻辑
  - 不需要事务写锁，可多进程并发读
  - 减少磁盘 I/O
- **限制**：禁止所有写操作（INSERT/UPDATE/DELETE/DDL），违反时抛出异常

### 5.4 网络模式选项 (`ServerDbOptions` 继承 `DbOptions`)

网络模式专属配置，包含 Binlog 相关设置：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|-------|------|
| EnableBinlog | Boolean | true | 是否启用 Binlog（网络模式默认启用） |
| BinlogMaxFileSize | Int64 | 256MB | 单个 Binlog 文件最大大小，达到后滚动到新文件 |
| BinlogRetentionDays | Int32 | 0 | Binlog 保留天数，0 表示不按时间清理 |
| BinlogMaxTotalSize | Int64 | 0 | Binlog 总磁盘空间上限（字节），0 表示不限制 |

## 7. 设计决策

### D1: 纯托管实现
不引入 native 依赖，保持跨平台一致性。所有代码使用纯 C# 实现，支持 .NET Framework 4.5+ 和 .NET Core/.NET 5-10。

### D2: 固定 8 种基础类型 + 扩展类型
- **8 种基础类型**：覆盖 90% 业务场景，保持类型系统简洁
- **扩展类型（GeoPoint/Vector）**：面向特定领域（LBS/AI），独立实现不影响核心
- **不支持自定义类型**：避免类型系统复杂化

### D3: NULL 值编码为 -1 长度前缀
- 所有变长类型（String/ByteArray）使用 4 字节长度前缀
- NULL 值编码为长度 -1（`[255, 255, 255, 255]`），无需额外 NULL 标记位
- 统一处理逻辑，简化编解码代码

### D4: 错误码分段管理
- 按模块划分错误码范围，便于定位问题来源
- 错误码全局唯一，避免冲突
- 预留充足的编码空间，支持未来扩展

### D5: 只读模式独立设计
- 只读模式不是"禁用写操作"的简单实现，而是深度优化的读性能模式
- 跳过 WAL 初始化、事务写锁等逻辑，减少资源消耗
- 支持多进程并发读，适用于数据分析场景

## 8. 未来扩展

### 8.1 自定义编解码器
允许用户注册自定义编解码器，支持扩展数据类型（如 JSON、XML、Protobuf）。

### 8.2 压缩编解码器
- 对 String/ByteArray 类型启用透明压缩（如 LZ4、Snappy）
- 在存储时压缩，读取时解压，应用层无感知
- 可按表/列配置压缩策略

### 8.3 加密编解码器
- 对敏感数据启用透明加密（如 AES-256）
- 支持静态加密（Encryption at Rest）
- 密钥管理与轮换策略

### 8.4 更多扩展数据类型
- **JSON**：存储为 BLOB，支持 JSON 路径查询
- **UUID**：128 位唯一标识符，16 字节存储
- **Enum**：枚举类型，映射为 Int32
