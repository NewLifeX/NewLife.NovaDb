# NewLife.NovaDb 索引架构

> 对应模块：N02（主键索引）、N04（二级索引）、N05（冷热分离）
> 关联模块：S03（页读写）、S04（页缓存）

---

## 1. 概述

NovaDb 使用 **SkipList**（跳表）作为核心内存索引结构，配合冷热分离策略，在内存占用与访问性能之间取得平衡。索引以 `.nvi` 文件持久化，启动时按需加载。

---

## 2. 跳表（SkipList）

### 2.1 结构

概率性多层有序链表，O(log N) 查找/插入/删除：

```
Level 3: HEAD ──────────────────── 50 ───────────── NIL
Level 2: HEAD ── 10 ─────────── 50 ── 70 ────── NIL
Level 1: HEAD ── 5 ─ 10 ── 20 ─ 50 ── 60 ─ 70 ── 90 ── NIL
Level 0: HEAD ── 5 ─ 10 ── 20 ─ 30 ── 50 ── 60 ─ 70 ── 90 ── NIL
```

### 2.2 参数

| 参数 | 值 | 说明 |
|------|---|------|
| MaxLevel | 16 | 支持 2^16 节点高效查找 |
| P | 0.5 | 每节点 50% 概率升层 |

### 2.3 节点内容

节点存储索引键 + 行定位信息（`PageId + Offset`），多层指针用于快速查找与范围扫描。

### 2.4 复杂度

| 操作 | 平均 | 最坏 |
|------|------|------|
| 查找 | O(log N) | O(N) |
| 插入 | O(log N) | O(N) |
| 删除 | O(log N) | O(N) |
| 范围扫描 | O(log N + K) | O(N + K) |

### 2.5 选择 SkipList 的理由

- 实现简单，代码量远小于 B+ 树
- 天然有序，范围扫描友好
- 并发友好（可扩展为 lock-free）
- 适合内存索引场景

## 3. 冷热分离

### 3.1 架构

```
┌──────────────────────────────────┐
│ 热索引 (HotIndexManager)         │  ← 最近访问的数据段
│ 完整 SkipList 节点                │     O(log N) 查找
├──────────────────────────────────┤
│ 冷索引 (ColdIndexDirectory)       │  ← 长期未访问
│ 稀疏目录（每 N 行一个锚点）        │     查询时按需加载
└──────────────────────────────────┘
```

### 3.2 热索引管理器（HotIndexManager）

管理热段，将最近访问的数据段完整加载到内存跳表：

- **热数据窗口**：默认 600 秒（`HotWindowSeconds`），窗口内保持热索引
- **冷淘汰阈值**：默认 1800 秒（`ColdEvictionSeconds`），超过则卸载
- **热段最大行数**：默认 100 万行
- **LRU 淘汰**：按最后访问时间排序，超时自动降级

段元数据（`IndexSegment`）：段 ID、是否热段、键范围（MinKey/MaxKey）、行数、最后访问时间。

### 3.3 冷索引目录（ColdIndexDirectory）

长期未访问的数据段仅保留稀疏目录：

```
锚点 0: Key=1    → PageId=0,  Offset=0
锚点 1: Key=1001 → PageId=5,  Offset=128
锚点 2: Key=2001 → PageId=10, Offset=256
...
```

- **锚点间隔**：默认每 1000 行
- **查找流程**：二分搜索稀疏目录 → 定位锚点 → 从锚点页顺序扫描
- **访问提升**：频繁访问的冷段自动提升为热段

### 3.4 冷热访问流程

```
查询 key=5000:
  1. 热索引查找 → 未命中
  2. 冷索引二分搜索 → 锚点(key=4000, offset=1234)
  3. MMF 加载锚点后 1000 行
  4. 构建热索引段(key=4000~5000)
  5. 缓存到热索引管理器
  6. 返回 key=5000 数据
```

### 3.5 内存效果

| 场景 | 全量索引 | 冷热分离 | 节省 |
|------|---------|---------|------|
| 1000 万行，查最近 10 万行 | ~1.5GB | ~15MB | 99% |
| 1 亿行，查最近 100 万行 | ~15GB | ~150MB | 99% |

## 4. 索引文件（.nvi）

### 4.1 多索引多文件

每个索引独立文件，便于独立加载、淘汰与重建：

- 主键索引：`{Table}.nvi`
- 二级索引：`{Table}_{IndexName}.nvi`

### 4.2 文件结构

`.nvi` 存储持久化的"目录 + 块/页"结构，**不是内存跳表的镜像**。内存 SkipList 从 `.nvi` 按需重建热点区域。

```
┌────────────────────────┐
│  FileHeader (32B)       │  FileType=Index
├────────────────────────┤
│  目录区域                │  稀疏锚点 → 块偏移
├────────────────────────┤
│  索引页序列              │
│  Page 0: 索引条目        │
│  Page 1: 索引条目        │
│  ...                    │
└────────────────────────┘
```

### 4.3 加载策略

1. 引擎启动：仅读取 `.nvi` 稀疏目录锚点（冷索引）
2. 首次查询某范围：从 `.nvi` 加载对应块到内存跳表（热索引）
3. LRU 淘汰时：热索引卸载回冷索引稀疏目录

## 5. 二级索引（P1）

### 5.1 类型

| 类型 | 说明 |
|------|------|
| 普通索引 | 加速 WHERE 条件查询 |
| 唯一索引 | 加速查询 + 唯一性约束 |
| 联合索引 | 多列组合，最左前缀匹配 |

### 5.2 存储

每个二级索引独立 `.nvi` 文件，索引条目记录：`(索引键, 主键值)`。查询时先走二级索引定位主键，再走主键索引取行数据（回表）。

### 5.3 维护

INSERT/UPDATE/DELETE 时同步更新关联的二级索引。唯一索引在写入前检查唯一性约束。

---

## 6. 设计决策

| # | 决策 | 要点 |
|---|------|------|
| D1 | SkipList 而非 B+ 树 | 实现简单、并发友好，适合内存索引 |
| D2 | 冷热分离 | 千万行仅查热点时内存 < 20MB |
| D3 | .nvi 非跳表镜像 | 持久化稀疏目录，按需重建热点 |
| D4 | 每索引独立文件 | 加载/淘汰/重建互不影响 |
