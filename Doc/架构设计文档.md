## 1. 系统架构概览

NewLife.NovaDb（诺娃数据库）是一个纯 C# 实现的混合数据库引擎,支持 .NET Framework 4.5+ 平台（含 .NET Core / .NET 5~10）,支持嵌入式/服务器双模部署。融合关系型、时序、消息队列、NoSQL(KV) 能力。

### 1.1 核心特性

- **纯 C# 实现**: 无 native 依赖,跨平台一致性
- **双模部署**: 嵌入式（进程内）/ 服务器模式（独立进程）
- **混合引擎**: Nova（关系型） + Flux（时序+MQ） + KV（键值）
- **MVCC 事务**: Read Committed 隔离级别,乐观并发控制
- **冷热分离**: 热数据内存索引,冷数据 MMF 按需加载
- **双日志系统**: WAL（崩溃恢复） + Binlog（主从复制）
- **SQL 引擎**: 支持 ~60% SQL92 标准,60+ 内置函数

### 1.2 整体架构图

```
┌──────────────────────────────────────────────────────────────────────┐
│                         应用层 (Application)                         │
├──────────────────────────────────────────────────────────────────────┤
│                     ADO.NET Provider (Client)                        │
│  NovaClientFactory │ NovaConnectionStringBuilder │ NovaDataAdapter   │
│  NovaConnection │ NovaCommand │ NovaDataReader │ NovaParameter       │
├──────────────────────────────────────────────────────────────────────┤
│                      SQL 引擎 (Sql)                                  │
│      SqlLexer → SqlParser → SqlAst → SqlEngine                      │
├─────────┬────────────┬──────────────┬────────────────────────────────┤
│ Nova    │ Flux       │ KV           │ 服务器 (Server)                 │
│ Engine  │ Engine     │ Store        │ NovaServer │ NovaProtocol   │
│ (关系型) │ (时序+MQ)   │ (键值)        │ NovaController│ NovaSession   │
├─────────┴────────────┴──────────────┼────────────────────────────────┤
│          事务层 (Tx)                 │       集群 (Cluster)            │
│ TransactionManager │ Transaction    │ ReplicationManager              │
│ RowVersion │ MVCC                   │ ReplicaClient │ NodeInfo        │
├─────────────────────────────────────┴────────────────────────────────┤
│                      WAL 日志层 (WAL + Binlog)                       │
│  WalWriter │ WalRecord │ WalRecovery │ BinlogWriter │ BinlogRecord   │

```


## 4. 设计决策记录（DDR）

### D1: 纯托管实现
不引入 native 依赖,保持跨平台一致性。

### D2: MMF 作为主要存储介质
利用操作系统的虚拟内存管理,突破物理内存限制。64 位系统虚拟地址空间 128TB,10G/100G 数据文件 OS 按需分页映射无压力。

### D3: SkipList 而非 B+ 树
SkipList 实现简单、并发友好,适合内存索引场景。

### D4: Read Committed 隔离级别
平衡一致性与并发性能,满足大多数业务场景。

### D5: 表级 WAL
每个表独立 WAL,减少并发竞争,简化恢复流程。WAL 用于崩溃恢复与持久性；完成 checkpoint 后可回收。只读模式不产生 WAL。

WAL/Checkpoint 的文件组织与生命周期细节见 [WAL与Binlog架构.md](架构/WAL与Binlog架构.md)。

### D6: 冷热分离索引
1000 万行表仅查热点时内存 < 20MB,冷数据自动卸载至 MMF。

### D7: SQL 子集策略
覆盖 ~60% 的 SQL92 标准,聚焦业务常用场景,不追求完整性。

### D8: Nova Engine 不分片
Nova Engine 以"表"为数据组织与管理单位,不做自动数据分片。分片带来的路由/边界管理复杂度通常大于收益；如需逻辑分表由上层应用完成。Flux Engine 面向时序/消息场景,可按时间维度进行数据分段组织。

相关文件布局与分段策略细节见 [存储架构.md](架构/存储架构.md)。

### D9: 表级并发控制
嵌入模式采用表级跨进程互斥,实现"一写多读",不同表互不影响。服务器模式单进程运行,MVCC 提供行级并发。

### D10: nova.db 二进制元数据
使用独立元数据存储保存数据库完整元数据（表/列/索引定义）,取代依赖系统表的方案。DDL 变更采用原子写入/替换策略,保证崩溃安全。

元数据的存储形态与更新流程细节见 [存储架构.md](架构/存储架构.md)。

### D11: 嵌入式与服务器存储统一
嵌入模式和服务器模式使用完全相同的文件布局。嵌入模式的数据库目录可直接拷贝到服务器模式使用,反之亦然。

### D12: KV 多表
允许一个数据库创建多个 KV 表（`ENGINE=KV`）,每个 KV 表用于不同业务场景,可配置独立的默认 TTL。

### D13: MQ 消息即时序行
Flux Engine 中,MQ 消息就是时序表的一行数据（MessageId + Fields）。Stream/Topic 名即时序表名,统一存储模型简化设计。

### D14: Binlog 独立于 WAL
WAL 是表级物理日志（崩溃恢复/持久性）,Binlog 是数据库级逻辑日志（主从复制/增量备份/审计）。两者职责正交、互不依赖。

### D15: Binlog 文件可安全手工删除
Binlog 以"文件段"为管理单位,段内自描述,便于运维进行磁盘空间回收。运维人员删除旧段用于释放空间不会导致引擎崩溃；代价是落后且未同步到该段的从节点需要全量重建或全量同步。

Binlog 的段组织、保留策略与删除语义细节见 [WAL与Binlog架构.md](架构/WAL与Binlog架构.md)。
