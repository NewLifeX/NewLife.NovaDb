### 任务 1：代码质量提升（P0，1 周）

**目标**：清理编译警告，提升代码健壮性

#### 准备工作
- 当前警告清单分类：Nullable 引用类型（~180）、缺失 XML 注释（~20）、xUnit 最佳实践（~6）

#### 执行步骤
1. **Nullable 引用类型警告处理**（约 180 个）
   - 客户端层（NovaConnection/NovaCommand/NovaDataAdapter/NovaParameter）：与 ADO.NET 基类的 nullability 不匹配
   - 引擎层（NovaTable.cs）：6 处解引用可能出现空引用
   - 测试层：#nullable 注释上下文外使用
   - **处理策略**：
     - ADO.NET 层：添加 `#pragma warning disable CS8765/CS8764/CS8600/CS8603`（基类设计限制）
     - 引擎层：增加防御性检查或使用 `!` 断言（经验证不可能为 null）
     - 测试层：统一启用 `#nullable enable`

2. **缺失 XML 注释补齐**（~20 个）
   - WalWriter 构造函数和 Dispose 方法
   - WalRecovery 构造函数
   - **按照编码规范 §7 补齐**

3. **xUnit 最佳实践修复**（~6 个）
   - 集合大小断言：改用 `Assert.Single`/`Assert.Empty`
   - 阻塞 Task 操作：改为异步测试方法

#### 验收标准
- ✅ 编译零警告（或警告数 < 10，已 pragma 抑制的除外）
- ✅ 所有 public/protected 成员有 XML 注释

### 任务 6：空间/向量索引实现（P1，3 周）

**目标**：为 GeoPoint/Vector 类型提供高效索引

#### GeoPoint 空间索引（2 周）

**准备工作**
- 研究 GeoHash 编码方案（推荐：精度 12，约 3.7cm）
- 或研究 R-Tree 实现（可参考 NetTopologySuite）

**执行步骤**
1. **GeoHash 索引实现**（推荐方案，实现简单）
   - 将 (Lat, Lon) 编码为 GeoHash 字符串
   - 作为二级索引存储（复用现有 SkipList 索引）
   - 实现 `WITHIN_RADIUS()` 查询优化
     - 计算边界框（BoundingBox）
     - 查询 GeoHash 前缀范围
     - 精确距离过滤

2. **R-Tree 索引实现**（备选方案，性能更优）
   - 实现 R-Tree 数据结构
   - 支持范围查询（BoundingBox）
   - 支持 KNN 查询（最近 k 个点）

**验收标准**
- ✅ `SELECT * FROM locations WHERE WITHIN_RADIUS(point, @center, 1000)` 查询时间 < 10ms（10 万行数据）
- ✅ 索引构建时间 < 1s（10 万行数据）

#### Vector 向量索引（1 周）

**准备工作**
- 研究 HNSW（Hierarchical Navigable Small World）算法（推荐）
- 或研究 IVF（Inverted File）算法（备选）

**执行步骤**
1. **HNSW 索引实现**（推荐方案）
   - 实现 HNSW 图结构
   - 支持 KNN 查询（Top-k 最相似）
   - 支持增量索引更新

2. **向量查询 SQL 扩展**
   ```sql
   SELECT id, COSINE_SIMILARITY(vec, @query_vec) AS score
   FROM embeddings
   ORDER BY score DESC
   LIMIT 10
   ```

**验收标准**
- ✅ KNN 查询（Top-10）时间 < 50ms（100 万向量，维度 1536）
- ✅ 召回率 > 95%（vs 暴力搜索）

### 任务 7：子查询支持（P1，2 周）

**目标**：支持 IN/EXISTS 子查询，提升 SQL 兼容性

#### 准备工作
- AST 层已预留 `SubqueryExpression`
- 需实现执行器层支持

#### 执行步骤
1. **IN 子查询**
   ```sql
   SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE status = 'paid')
   ```
   - 执行子查询获取结果集
   - 转换为 IN 条件过滤

2. **EXISTS 子查询**
   ```sql
   SELECT * FROM users WHERE EXISTS (SELECT 1 FROM orders WHERE orders.user_id = users.id)
   ```
   - 对每行外层数据执行子查询
   - 返回 True/False

3. **优化：子查询结果缓存**
   - 非相关子查询（子查询不依赖外层）：仅执行一次
   - 相关子查询：每行外层数据执行一次（可优化为 JOIN）

#### 验收标准
- ✅ 支持 IN/EXISTS 子查询语法
- ✅ 非相关子查询仅执行一次
- ✅ 性能测试：子查询 vs JOIN（差距 < 2x）

### 任务 8：Hash Join 优化（P1，1 周）

**目标**：优化 JOIN 性能，替代 Nested Loop Join

#### 准备工作
- 当前仅支持 Nested Loop Join（O(n*m)）
- Hash Join 适用于等值 JOIN（O(n+m)）

#### 执行步骤
1. **Hash Join 实现**
   - 构建哈希表（小表）
   - 探测哈希表（大表）
   - 支持 INNER/LEFT/RIGHT JOIN

2. **优化器：自动选择 JOIN 策略**
   - 小表 < 1000 行：Nested Loop Join
   - 大表：Hash Join
   - 有索引：Index Nested Loop Join（未来）

#### 验收标准
- ✅ Hash Join 性能 vs Nested Loop Join：10x+ 提升（大表场景）
- ✅ 自动选择 JOIN 策略（基于表大小）

## 七、风险管理与应对策略

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| **性能不达标** | 无法替代现有数据库 | 中 | 尽早建立基准测试，持续优化热点路径 |
| **文件格式不兼容** | 升级时数据丢失 | 低 | 建立迁移工具，版本号严格管理 |
| **主从同步延迟过高** | 读写分离效果差 | 中 | 优化 Binlog 传输，支持批量复制 |
| **空间/向量索引召回率低** | AI 场景不可用 | 中 | 提供可调参数（精度 vs 速度权衡） |
| **XCode 集成兼容性问题** | 现有项目无法迁移 | 低 | 完善方言适配，提供迁移脚本 |
| **开源社区参与度低** | 缺乏外部贡献 | 高 | 完善文档，举办技术分享会，参与开源活动 |

## 十、协作建议

### 与 AI 协作的最佳实践

1. **任务拆分原则**
   - 单个任务不超过 1 周（AI 上下文限制）
   - 每个任务有明确的输入、输出、验收标准
   - 优先完成独立模块（减少依赖）

2. **代码生成策略**
   - AI 生成代码后，人工 Review 必不可少
   - 关键算法（如索引）需提供伪代码或参考实现
   - 性能敏感代码需基准测试验证

3. **文档驱动开发**
   - 每个任务先完善设计文档（`Doc/` 目录）
   - AI 根据文档生成代码
   - 文档与代码保持同步

4. **增量交付**
   - 每完成一个任务，立即提交 PR
   - 通过 CI 验证后合并
   - 避免大批量代码积压
